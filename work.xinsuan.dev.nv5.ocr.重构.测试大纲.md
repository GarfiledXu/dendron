# OCR C++ 重构版本验证与全链路测试大纲

**核心准则**：在同等配置与同一批输入图像下，新版 C++ 节点的输出状态（Status）、得分（Score）、字符串（String）与内存表现，必须与旧版 C 语言节点 100% 对齐。绝不允许任何业务特性的静默降级。

---

## 阶段一：业务功能等价性验证 (Functional Equivalence / 黑盒)

本阶段依据上位机 UI 的交互入口进行穷举测试，确保对客业务场景无缝兼容。

### 1. 字符串模式 (String Mode)

* **[ ] 基础识别与字符集**
  * **测试点**：配置读取大写/小写/数字/特殊符号。
  * **验证**：算法单字提取准确率、总得分计算是否与老版一致。
* **[ ] 长度约束与裁断**
  * **测试点**：配置 `min_char_count` 和 `max_char_count`。
  * **验证**：当识别字符数超出或不足时，节点是否能正确抛出 NG 状态；截断逻辑是否正确。
* **[ ] 控制字符与过滤**
  * **测试点**：启用 `ignore_sep_cfg`（如忽略 `-`、`/` 等分隔符）。
  * **验证**：输出的 `final_str` 是否成功去除了配置的符号，且 `char_list` 坐标与原图字符依旧对应。

### 2. 日期模式 (Date Mode)

* **[ ] 基础格式校验**
  * **测试点**：配置不同的日期排布（YYYYMMDD, DDMMYYYY, HHMM 等）。
  * **验证**：提取的数字流能否正确映射为日期/时间格式；格式不符时是否正确判 NG。
* **[ ] 日历同步与偏移补偿 (Calendar Sync)**
  * **测试点**：测试 `month_offset`, `day_offset` 等补偿参数。
  * **验证**：注入基准时间后，偏移后的时间计算是否正确。

### 3. 公共特性 (Common Features)

* **[ ] 几何区域与掩码 (ROI & Masks)**
  * **测试点**：配置 1 个主搜索框 + 多个屏蔽区/附加区。
  * **验证**：底层 `mask_img` 渲染是否正确（屏蔽区内的干扰文字不被识别）；上位机看到的 JSON 中 `matched_region` 和 `assistive_region` 数组无缺失。
* **[ ] 位置修正联动 (Position Adjust)**
  * **测试点**：关联 M00 位置修正节点，故意倾斜/平移测试工件。
  * **验证**：OCR 是否成功接收仿射变换矩阵，ROI 搜索框能否跟随工件正确发生旋转与偏移。

## 阶段二：核心算法数据驱动测试矩阵 (Data-Driven Unit Testing / 白盒)

本阶段针对极其容易产生 Corner Case 的底层逻辑（字符串过滤、日历跨年跨月进位），使用参数化测试（Parameterized Tests，如 GoogleTest `INSTANTIATE_TEST_SUITE_P`）进行 100% 覆盖。

### 1. 字符串匹配逻辑超集 (String Match Matrix)

| 测试场景分类 | 算法输出(Raw String) | 主控字符(Target) | 约束规则配置 (Rules) | 预期判定 (Status) | 预期输出字符 (Final String) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **完美匹配** | `ABC123` | `ABC123` | 无特殊约束 | **OK** | `ABC123` |
| **长度截断 (超限)** | `ABCD12345` | 无 (泛读) | `max_char=6` | **OK** | `ABCD12` |
| **长度不足 (短缺)** | `AB` | 无 (泛读) | `min_char=4` | **NG** | `AB` |
| **忽略分隔符 (纯净匹配)**| `2026-06/18` | `20260618` | `ignore_sep=['-', '/']` | **OK** | `20260618` |
| **忽略分隔符 (带乱码)** | `A*B.C` | `ABC` | `ignore_sep=['*', '.']` | **OK** | `ABC` |
| **大小写容错** | `aBcD` | `ABCD` | 开启大小写不敏感 | **OK** | `aBcD` (或转大写) |
| **剔除屏蔽位** | `123X456` | 无 | 屏蔽第4位 | **OK** | `123456` |

## 阶段三：资源与稳定性压测 (Resource & Stability) [高优]

由于引入了 C++ STL 与复杂的对象生命周期，必须防范内存泄漏。

* **[ ] 节点生命周期清理 (Init / Deinit)**
  * **测试点**：在上位机中反复添加、删除 OCR 节点（操作 100 次）。
  * **验证**：观察系统可用内存。确保析构时 `cJSON` 彻底释放，`nvs_image_destroy` 成功销毁了复用的 `mask_img`。
* **[ ] 动态内存池压测 (Runtime Leak Check)**
  * **测试点**：内部触发模式下，开启极速节拍连续运行 1 万帧。
  * **验证**：通过 `top` 或 `free` 监控内存水位。确保 `proc_result_pack_` 中每帧挂载的 JSON 字符串与 RAW STR 没有引发堆栈溢出。
* **[ ] 持久化存储与读写 (Storage)**
  * **测试点**：保存模板、样本（`ocr_schemes`）或配置参数至 Flash/文件系统。
  * **验证**：断电重启后，反序列化加载的参数是否完好，不发生数据截断或运行时异常（如段错误）。

## 阶段四：通信契约与节拍性能 (Protocol & Performance) [中优]

保障对下（PLC）和对上（UI）的接口协议无缝衔接。

* **[ ] 工业总线载荷 (Fieldbus Payload)**
  * **测试点**：不接上位机，仅通过 PLC 交互。
  * **验证**：`priv_.fb_tool_payload` 打包的字节序、阈值和得分偏移量是否与 GSDML 文件定义严格对齐。
* **[ ] 节拍耗时核对 (Cycle Time)**
  * **测试点**：同一张图，老版本与新版本的 `process` 整体耗时对比。
  * **验证**：C++ 版本的 `cost_time_avg` 不得产生明显的性能退化（如差异 > 10% 需排查深拷贝开销或 JSON 序列化瓶颈）。
* **[ ] 极端异常注入 (Chaos Testing)**
  * **测试点**：传入全黑图、尺寸为 0 的异常图、清空所有 ROI、断开位置修正前置节点。
  * **验证**：程序不崩溃，能优雅返回 `NVS_OUTPUT_NG` 并在底层日志中正确打印拦截报错。

## 阶段五：系统级协议与集成验证 (System Protocol & Integration)

本阶段脱离上位机 UI 依赖，直接从整机对外通信端口（网口/串口/总线）抓包，验证 C++ 节点在真实工业环境下的通信契约与容错能力。

### 1. 工业总线协议验证 (Fieldbus / PLC)

* **[ ] GSDML Payload 内存对齐测试**
  * **测试点**：通过 `pack_fieldbus_payload` 写入的 `priv_.fb_tool_payload`。
  * **验证**：不接上位机，仅连接 PLC（或使用总线模拟器）。抓包核对 OCR 输出的 `Score`、`Threshold`、`String` 在字节流中的起始地址、大小端序（Endianness）是否与老版本严格一致。
* **[ ] 超长载荷截断防线**
  * **测试点**：让 OCR 识别一段长度为 100 字节的超长字符串。
  * **验证**：总线 Payload 通常有严格的长度限制（如 64 或 128 字节）。验证超长字符串是否被安全截断，且**绝对不能引发内存越界覆盖后面的判定标志位**。
* **[ ] 工具状态字联动测试 (Status Words)**
  * **测试点**：触发一次完美的 OK 识别，和一次 NG 识别。
  * **验证**：检查总线报文中的全局状态位（如 `FB_RES_TOTAL_JUDGE_OK`）以及该特定工具的 Bit 位（如 `T00` 对应的标志位）是否能跟随 C++ 引擎的 `report.status` 准确翻转。

### 2. User TCP 自定义协议验证 (User TCP)

* **[ ] 双模式报文兼容性**
  * **测试点**：切换系统配置 `sys_opts.user_tcp_detailed_fmt`（简略模式 0 vs 详细模式 1）。
  * **验证**：核对 `NVS_NODE_USR_RES_KEY_FULL_RAW_STR` 生成的逗号分隔字符串。详细模式必须包含前缀、工具类型 `OC`、分数、定长字符串等；简略模式必须只包含极简判定。
* **[ ] 特殊字符编码转义**
  * **测试点**：识别包含逗号 `,` 或回车换行 `\r\n` 的乱码工件。
  * **验证**：User TCP 强依赖逗号分段和换行符结尾。验证 C++ 引擎在拼接 `raw_str` 时，是否对识别出的逗号进行了安全转义或过滤，防止 TCP 接收端粘包或分段错乱。

### 3. 错误上报与异常防线 (Error Reporting & Alarms)

* **[ ] 算法错误透传 (Algo Error Context)**
  * **测试点**：构造极端错误环境，如：未注册模板（强行触发 RUN）、算法句柄初始化失败、内存耗尽（传入极大分辨率的 `mask_img`）。
  * **验证**：C++ 节点必须调用 `nvs_push_algo_error`。验证错误码和报错提示能否成功推送到整机的报警队列，并在外部设备的报警灯/错误日志中体现。
* **[ ] 硬件外设联动 (Output Assign)**
  * **测试点**：在物理机器上，将 IO 输出引脚绑定给 OCR 节点的 OK/NG 信号。
  * **验证**：当 C++ 节点算出 NG 时，核对框架底层的 `nvs_process_output_assign_trigger` 能否正确拉低/拉高物理引脚电平。

### 4. 状态机与缓存流转 (State Machine & Context)

* **[ ] 运行时状态切换安全**
  * **测试点**：在连续高速触发（高频发图）的过程中，突然下发配置要求节点进入 `LEARN` 或 `IDLE` 状态。
  * **验证**：C++ 的资源锁或生命周期必须稳固。验证引擎是否能优雅地拒绝处理当前帧或安全完成切换，不发生死锁或野指针访问。
* **[ ] 缓存清理与脏数据防御**
  * **测试点**：先跑成功一次 A 工件（得到 OK 和字符串），然后遮挡镜头跑 B 工件导致识别失败。
  * **验证**：B 工件的 JSON、RAW STR、TCP 报文和总线 Payload 中，绝不能残留 A 工件的旧字符串或分数（即 `priv_.state_cache` 每次必须彻底更新或清零）。

## 判定逻辑-功能项划分

### 字符串模式

* 设定-扩展功能-设置分隔符
  * 参数：开关：无效
  * 参数：开关：有效 + 组合字符（[*] [/] [-] [*] [:] [汉字：年月]）当然也有可能是其他自定义的汉字和符号
  * 功能定义：开关有效时，对主控字符和结果字符都有影响，均会过滤；属于字符匹配前的多种字符过滤
* 设定-读取设定
  * 参数：读取字母：无效、大写、小写、大小写
  * 参数：数字读取：有效/无效
  * 参数：符号读取：有效/无效
  * 参数：字符读取：有效/无效
  * 功能定义：对结果字符有影响，会对结果进行过滤；属于字符匹配前的结果字符过滤
* 设定-判定条件-主控字符的统配符
  * 参数：?:全部，$:字母，%:数字，@:符号，!:大写字母，&小写字母
* 设定-判定条件-仅判断字符数（该功能和通配符的只会二选一）
  * 参数：最大字符数量和最小字符数量

### 三期模式

* 设定-扩展功能-设置分隔符
  * 参数：开关：无效
  * 参数：开关：有效 + 组合字符（[*] [/] [-] [*] [:] [汉字：年月]）当然也有可能是其他自定义的汉字和符号
  * 功能定义：开关有效时，对主控字符和结果字符都有影响，均会过滤；属于字符匹配前的多种字符过滤

* 设定-字符读取-日期格式
  * 参数：年月日、日月年、月日年、年月、月年、时分、小时、分钟
  * 功能定义：在日历同步功能开启后，将影响主控字符的生成（根据当前时间配置生成）：根据时间戳+日期格式+从原主控字符中提取到的分隔符 合成新的主控字符串；
* 设定-判定条件-日历同步
  * 参数：数值：补偿（年月日补偿）
  * 参数：数值：运行误差（分钟）
  * 参数：开关：十位中的0（月日时分）
  * 参数：补偿计算顺序（月->日->分 or 分->日->月）
  * 参数：一个月补偿设定（当月末/下月初/顺延）
  * 参数：时间显示方式（24小时/12小时）
  * 参数：开关：调节到1个月前

  * 从主控字符串中提取出对应的分隔符

#### 暂时不实现
