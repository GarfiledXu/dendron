**文档编号**：NVS-ARCH-042  
**模块名称**：`nvs_pre_rule_check`  
**架构形态**：主动查询式预检机制 (支持双轨模式)

---

## 一、 模块概述和设计原则

### 1.1 功能定义

作为**下位机**的规则验证服务，负责接收来自**上位机**的查询请求。接口支持**“双轨模式”**：

1. **配置试运行 (Dry-Run)：** 在实际保存参数前，附带测试参数请求校验，下位机评估参数合法性但不写入实际内存。
2. **运行态校验 (Status Check)：** 在实际触发运行前，基于当前系统环境、物理资源以及设备拓扑状态，评估是否满足执行条件。

### 1.2 价值体现

核心在于将“防呆”与“防错”逻辑全面下沉至下位机。通过统一的事前查询接口拦截无效动作：

1. **避免上位机硬编码规则（配置试运行）：** 上位机在保存参数前附带测试负载发起校验。下位机自动联动底层硬件能力集进行评估，并返回报错字段的具体路径。上位机彻底免去了硬编码硬件差异的困扰，也避免了繁琐的冗长交互。
2. **系统资源前置评估：** 在执行保存文件或添加样本等消耗型动作前，接口会预先比对真实的存储空间或内存剩余量，防止执行中途因资源耗尽引发写入失败或系统异常。
3. **拓扑与依赖关系验证（运行态校验）：** 联动全局能力集及设备自身状态，在动作下发前完成数据流闭环、依赖状态就绪等跨节点逻辑校验，阻止无效配置引发运行时错误。

### 1.3 核心设计原则

1. **状态只读：** 不修改下位机内部数据，也不触发真实硬件动作，仅进行只读计算并返回结果。
2. **职责划分：** 各层级实体（SYSTEM、PROG、NODE）仅负责维护和校验与其自身直接相关的配置项与状态数据，模块间互不越权。

## 二、 角色定义和执行链路

### 2.1 角色定义

* **上位机 (Upper Machine)：**
  * **前端控制端：** 触发方。负责在执行特定交互前发起查询请求，并根据下位机透传的结果进行控件提示或全局警告。
* **下位机 (Lower Machine)：**
  * **NCM (通信模块)：** 负责接收网络数据，拆解定界符并提取指令与 JSON 负载，向下分发。
  * **NDM (设备管理模块)：** 接收业务指令，作为下位机的总控，调用预检引擎的统一入口。
  * **预检调度器 (Pre-check Dispatcher)：** 检查流程的统筹方。负责初始化错误收集上下文，解析请求层级，并寻址路由到对应的具体回调函数。
  * **Prog/Node 实体 (程序与节点)：** 实际的校验规则执行方。持有当前状态数据与内部限制条件，通过注册的 C 函数接口执行比对。

### 2.2 执行链路

1. 上位机下发 `Pre_Rule_Check` 业务查询指令。
2. 下位机 NCM 完成底层协议解析，将提取出的**检查层级（Scope）、目标标识（Target ID）及可选的测试参数（Payload）**移交 NDM。
3. NDM 唤起 `nvs_pre_rule_check_run` 接口，传入上述参数。
4. 预检调度器创建上下文，根据层级路由到 Prog 管理器或具体的 Node 工具注册的检查回调。
5. 底层模块结合**测试载荷（若有）、当前的设备真实环境、黑板配置及全局能力集**运行比对，将产生的错误项压入校验上下文。
6. 预检调度器收集所有错误项，组装为标准业务响应（ACK）返回给上位机。

## 三、 检查目标和结果的层级划分

### 3.1 检查目标层级 (Scope)

* **SYSTEM (0)：** 系统级。评估设备全局硬件环境（如存储空间）与软件能力基线。
* **PROG (1)：** 程序级。评估程序内部的拓扑关系、工具节点配额上限、组件数据流的连通性。
* **NODE (2)：** 节点级。评估单一工具内部参数的具体枚举范围，以及独立状态机的就绪情况（如模板是否已学习）。

### 3.2 结果严重等级与动作定义 (Level)

* **PASS (0)：** 检查通过，数据合规，允许放行。
* **WARN (1)：** 警告。检测到逻辑风险，但不强制阻断动作指令的下发，上位机可选择性提示用户。
* **FATAL (2)：** 致命。检测到严重越界、资源不足或拓扑断裂。此状态意味着数据不可用，下位机建议拒绝后续动作，上位机应强制拦截。

## 四、 上下位机交互时序与负载定义

### 4.1 交互时序

上位机在提交配置保存或主动触发运行前发起预检请求，阻塞等待下位机返回 ACK 报文。确认返回结果中无 `FATAL` 级别错误后，方可继续下发真实的写入或运行指令。

### 4.2 负载 (Payload) 字段定义

基于系统已确立的 `指令-#JSON负载[CR]` 底层通信协议，本模块重点对 JSON Payload 的业务字段进行定义。

**请求字段 (Request Payload)：**

* `scope`: 整型 (0: System, 1: Prog, 2: Node)。
* `target_id`: 字符串。对应程序 ID 或节点 ID（检查 System 级时为空）。
* `payload`: JSON 对象。
  * **轨道 A：** 传入具体的参数对象，下位机执行“配置试运行”校验。
  * **轨道 B：** 传 `null` 或 `{}`，下位机执行“当前运行态”校验。

**应答字段 (ACK Payload)：**

* `status`: 整型 (0 表示网络与协议层处理成功，非校验结果)。
* `errors`: JSON 数组。包含所有被拦截的异常项。
  * `level`: 整型 (1: WARN, 2: FATAL)。
  * `err_code`: 整型，全局错误码。
  * `err_source_id`: 字符串。指出是哪个节点或模块导致了阻塞（主要用于轨道 B 状态查询）。
  * `err_path`: 字符串。导致校验失败的具体 JSON 字段路径（主要用于轨道 A 配置试运行，供前端精准标红输入框）。
  * `err_msg`: 字符串，原因描述文本。

### 4.3 负载数据示例

**场景一：参数配置校验 (轨道 A - 带有 payload)**

```json
// 请求
{"scope": 2, "target_id": "ocr_node_01", "payload": {"threshold": 300}}

// 应答
{
  "status": 0,
  "errors": [
    {"level": 2, "err_code": 1004, "err_source_id": "ocr_node_01", "err_path": "threshold", "err_msg": "Exceeds hardware limits."}
  ]
}
