# NVS 系列设计文档生成提示词（风格模板）

请帮我生成一篇偏“嵌入式架构设计风格”的技术设计文档，要求遵循以下文档风格与排版规范：

---

# 一、整体风格要求

- 文档风格偏：
  - 嵌入式系统设计
  - 模块架构设计
  - 上下位机协议设计
  - 工程实现导向

- 不要写成：
  - 产品宣传稿
  - PPT 风格
  - 教学教程
  - AI 生成式长篇散文

- 内容要求：
  - 简洁
  - 工程化
  - 结构清晰
  - 强调职责边界
  - 强调模块关系
  - 强调设计目的
  - 强调执行链路

- 避免：
  - 过度口语化
  - 过度文学化
  - 大段空洞描述
  - 重复性总结
  - 冗长概念解释

---

# 二、文档结构风格

文档整体采用：

```text
章节 -> 小节 -> 要点
```

形式。

推荐结构：

# 1. 模块概述与设计原则
## 1.1 功能定义
## 1.2 价值体现
## 1.3 设计原则

# 2. 角色定义与执行链路
## 2.1 角色定义
## 2.2 执行链路

# 3. 数据结构 / 协议定义

# 4. 内部实现设计

# 5. 典型场景

# 6. 总结

---

# 三、段落风格要求

每个小节尽量遵循：

```text
一句核心定义
↓
若干要点
↓
必要时补充说明
```

例如：

```markdown
### 1）运行状态检查

用于检查目标对象当前真实运行状态。

包括：

- Flash 资源
- Tool 状态
- 拓扑关系
- Runtime 状态

该类问题通常属于：

- 状态级错误
- 资源级错误
```

不要出现：

- 超长自然语言段落
- 连续大篇解释
- 情绪化措辞

---

# 四、术语风格要求

术语表达偏：

```text
工程术语 + 架构术语
```

例如：

- 规则解释权
- 能力集
- 拓扑闭环
- 状态机
- 运行态
- 配置负载
- 动态防呆
- 调度器
- 路由
- 上下文
- Payload
- Dispatcher
- ACK

避免：

- 市场化表达
- 宣传式表达
- 口号式表达

---

# 五、列表与排版规范

优先使用：

- 短句
- Bullets
- 表格
- 分层标题

避免：

- 大段连续正文

推荐：

```markdown
- 功能 A
- 功能 B
- 功能 C
```

而不是：

```markdown
系统首先会...
随后会...
最终会...
```

---

# 六、设计原则表达风格

设计原则部分重点强调：

- 职责边界
- 解耦
- 无副作用
- 规则归属
- 模块层级
- 统一入口
- 可扩展性
- 运行态与配置态区分

表达风格偏：

```text
原则 + 原因
```

例如：

```markdown
### 1）无副作用

预检过程仅进行规则评估：

- 不修改内部数据
- 不保存配置
- 不改变状态机
```

---

# 七、架构描述风格

架构描述强调：

- 调用链路
- 模块职责
- 数据流
- 调度关系

避免泛泛描述。

优先描述：

```text
谁调用谁
谁负责什么
谁拥有解释权
谁负责路由
```

---

# 八、协议设计风格

协议部分要求：

- 表格化
- 字段职责明确
- 区分 Request / ACK
- 明确字段意义
- 明确特殊字段用途

例如：

| 字段 | 类型 | 说明 |
|---|---|---|

并补充：

- 空值语义
- 特殊字段意义
- UI 联动关系

---

# 九、代码示例风格

代码示例要求：

- 偏工程伪代码
- 强调核心逻辑
- 不需要完整实现
- 注释简洁明确

避免：

- 教学式注释
- 无意义样板代码

---

# 十、PlantUML 风格

如果需要时序图或结构图：

- 风格简洁
- 不要过度美化
- 强调层级与调用关系
- 偏工程图而非展示图

推荐：

- Dispatcher
- Scope
- Node
- Runtime
- Payload

这类命名。

---

# 十一、总结风格

总结部分不要写成：

```text
宏大叙事
```

而是强调：

- 本模块解决了什么问题
- 职责如何划分
- 为什么这样设计
- 带来了哪些工程收益

篇幅控制简洁。

---

# 十二、Markdown 输出要求

请输出：

- 纯 Markdown 源码
- 不要渲染说明
- 不要额外解释
- 不要使用 markdown 代码块嵌套错误

最终请将全文包裹在：

四重反引号 markdown 代码块

中输出，避免内部代码块冲突。
